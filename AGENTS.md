# Codex项目规则｜V6

## 项目目标

这是“骑不快的ZZ（大熊说车）”的性格选车与装备匹配工具。核心不是堆参数，而是帮助普通摩友理解：保护、舒适、操控、美观、维护和成本之间不存在全能方案。

## 不允许改坏

1. 主选车保持一题一屏，不改成长表单。
2. 主选车仍为12轮，装备问题放独立扩展包。
3. 装备扩展包固定八类：头盔、手套、护具/骑行服、骑行靴、三箱/尾箱、射灯、蓝牙耳机、防盗/盗抢保障。
4. 每项至少保留：真实手感、外观建议、取舍指标、代价、优先项、错误买法、核对清单。
5. 每项完成后先显示3:4提醒图片，再进入下一项。
6. 不增加橱窗链接、购买按钮、跳转按钮、商品卡、手机号收集或登录。
7. 橱窗只允许作为一句自然口播，不得压过装备判断。
8. UI保持深蓝、夕阳橙、复古机车海报感。
9. 答案只保存在浏览器本地。
10. 不编造认证、实验数据、车型参数或保障条款。

## 手套特别规则

- 必须区分城市、街道运动、旅行和赛道手套。
- 赛道级不能写成全面更好。
- 必须说明赛道长护腕、厚皮、预弯、掌根滑块和护壳可能牺牲：
  - 舒适
  - 穿脱
  - 低速灵活
  - 握把细腻感
- 可以使用用户能听懂的“握屎感”，但同时解释原因：掌心厚度、版型、预弯、尺码、护壳和磨合。
- 街道运动型应被描述为：保护高于轻量通勤，舒适和抓握通常优于纯赛道。

## 其他装备细节规则

### 头盔

同时考虑头型、夹点、气动、风噪、重量、通风、眼镜/耳机和外观。中国道路头盔合规信息需要核对当前有效标准，不按材料纹路直接推荐。

### 护具/骑行服

区分撞击保护、外层耐磨、护具跑位、穿戴率、气候和版型美观。

### 骑行靴

区分脚踝抗扭、胫骨覆盖、步行、换挡脚感、防水和外观。赛道靴不能按普通鞋舒适度评价。

### 三箱

区分单尾箱、硬三箱、软包和尾包。计算容量、箱体自重、支架、载荷、风阻、车宽、重心和整车比例。

### 射灯

区分截止近场、远射、泛光和组合光型。保留电路余量、保险、线径、接地、防水、眩光和当地要求。

### 蓝牙耳机

区分单人、双人、小车队、多人组网。考虑重连、耳槽、扬声器厚度、操作、风阻和头盔侧面美观。

### 防盗

采用停车环境、机械锁、定位、报警、车罩和保障的分层方案。不能承诺百分百防盗、找回或赔付。

## 配置位置

提醒图和自然口播统一配置：

```text
src/accessories.js -> REMINDER_CONFIG
```

不允许重新加入 `shopUrl`。

## 每次完成必须运行

```bash
npm run check
npm run build
```

至少检查：

- 8个模块均有5题以上
- 赛道手套结果出现舒适/握持代价
- 街道运动手套不被写成轻薄无保护
- 结果页指标条正常显示
- 手机390px无横向溢出
- 源码中不存在 `shopUrl`、`open-shop`、`点击购买`、`立即购买`

## V6.2图片与平台入口规则

- 具体车型真实图优先来自公开车型来源页，必须保留来源URL。
- 没有真实图时使用本地类别方向图，并明确标注为示意图。
- 不得从类别方向图推断具体车型外观。
- 平台按钮只允许生成搜索入口，不允许硬编码具体商品、佣金参数或橱窗链接。
- 价格、库存、销量、认证和评价均属于实时信息，页面必须提示用户再次核验。
- 头盔品牌只能作为候选池，不得写成“某品牌全部安全”或“某品牌所有型号都推荐”。
